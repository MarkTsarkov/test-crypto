# Crypto Withdrawal API

```bash
docker compose up -d
```

При запуске миграции накатятся автоматически через отдельный контейнер-мигратор на базе goose. Миграция также засеивает тестового пользователя с балансом 1000 USDT.

---

## Принятые решения

### Идемпотентность

Главная сложность — гарантировать, что два одновременных запроса с одинаковым `Idempotency-Key` не создадут два вывода средств. Решение строится на двух уровнях:

1. **`pg_advisory_xact_lock`** — берётся в начале транзакции на пару `(idempotency_key, user_id)`. Второй параллельный запрос встанет в очередь и дождётся завершения первого, после чего обнаружит уже готовый ответ в кэше.
2. **Таблица `idempotency_keys`** — хранит хэш тела запроса и закэшированный ответ. Если ключ уже использовался с другим телом — возвращается `422 Unprocessable Entity`.

### Двухфазный вывод

Вывод сознательно разделён на два запроса: создание (`pending`) и подтверждение (`complete`). Это имитирует реальный процесс, где между отправкой в блокчейн и фиксацией результата есть временной разрыв. Запись в `ledger_entries` происходит только при подтверждении — это гарантирует, что в бухгалтерском реестре окажутся только фактически завершённые операции.

### Транзакции

Вся цепочка при создании вывода — блокировка ключа, проверка идемпотентности, проверка баланса, создание записи — выполняется в одной транзакции. Транзакция передаётся через `context.WithValue`, чтобы репозиторий не знал о её существовании явно.

### Комиссия сети

Захардкожена в `1` единицу валюты. При проверке баланса требуется `amount + 1 ≤ balance`. В реальном сервисе это вычислялось бы динамически через внешний источник курсов.

### Аутентификация

Токен сравнивается через `subtle.ConstantTimeCompare` — защита от timing-атак. Значение токена читается из переменной окружения `USER_TOKEN`. Это намеренное упрощение: в боевом сервисе здесь был бы полноценный JWT с верификацией подписи.

## Как запустить

**Требования:** Docker, Docker Compose

```bash
# 1. Запустить всё
docker compose up -d

# 2. Проверить, что всё поднялось
docker compose ps
```

Сервис будет доступен на `http://localhost:8080`.

Все переменные окружения уже прописаны в `.env`. Менять ничего не нужно.

## Как протестировать

В корне проекта лежат готовые файлы для Postman:

- `crypto-withdrawal.postman_collection.json`
- `crypto-withdrawal.postman_environment.json`

**Шаги:**

1. Импортировать оба файла в Postman (File → Import)
2. Выбрать окружение `Crypto Withdrawal - Local`
3. Открыть окружение и вставить токен из `.env` (значение `USER_TOKEN` без префикса `Bearer `) в поле `user_token`
4. Запускать запросы по порядку: **1 → 2 → 3**

`Idempotency-Key` генерируется автоматически перед каждым запросом. После создания вывода `withdrawal_id` сохраняется в переменные коллекции и подставляется в подтверждение автоматически.

Тестовый пользователь (`12345678-1234-4123-b234-123456789012`) уже создан миграцией с балансом **1000 USDT**.

## Как бы я улучшил решение

1. **Комиссия сети динамически** — сейчас захардкожена в 1. Нужен отдельный сервис или таблица с актуальными fee по каждой сети.

2. **Асинхронная отправка в блокчейн** — сейчас подтверждение вывода ручное (отдельный HTTP-запрос). В реальности нужна очередь (Kafka / RabbitMQ): воркер забирает `pending`-выводы, отправляет в сеть, получает хэш транзакции и подтверждает. Это развязывает API от задержек блокчейна.

3. **Вычитание баланса** — сейчас баланс только проверяется, но не списывается. В реальной системе при создании вывода нужно блокировать (резервировать) сумму, а списывать только при подтверждении.

4. **Нормальный JWT** — текущая проверка токена примитивная. Нужна верификация подписи, проверка `exp`, возможно scope-based авторизация.

5. **Graceful shutdown** — сервис сейчас останавливается жёстко. При завершении нужно дождаться окончания активных запросов, чтобы не оборвать транзакции на полуслове.

6. **Health check** — отдельный эндпоинт `/health` с проверкой коннекта к БД, чтобы оркестратор знал, когда сервис реально готов к трафику.

7. **Метрики и трейсинг** — OpenTelemetry для отслеживания латентности по каждой фазе (lock, balance check, insert) и понимания узких мест под нагрузкой.

8. **Валидация адреса назначения** — сейчас `destination` принимает любую строку. Нужна проверка формата адреса в зависимости от валюты (USDT TRC-20 vs ERC-20 — разные форматы).
